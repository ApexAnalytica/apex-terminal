01. System Overview

APEX Terminal is a causal intelligence platform for analyzing systemic risk in critical infrastructure supply chains. It combines interactive 3D graph visualization with four specialized causal reasoning engines, dual-LLM copilot integration, and an epoch-based cascade replay system.

Disconnected Node Detection

After any graph change (import, scissors), the renderer computes connected components via BFS. Nodes outside the largest connected component render as grey. When new edges bridge a floating cluster to the main graph, those nodes transition to full color automatically.

05. Core Engines

APEX Terminal has four reasoning modules, plus two additional analysis tools (Ablation and Interdiction). All engines are accessed through a unified EngineProvider interface, currently implemented by LocalProvider (client-side). A future RemoteProvider will swap transport (FastAPI/gRPC) without changing calling code.

Engine Provider Interface

  src/lib/engines/
  +-- engine-interface.ts    EngineProvider interface (common contract)
  +-- local-provider.ts      LocalProvider (client-side implementation)
  +-- index.ts               Registry (getEngineProvider() singleton)

  Methods:
  +-- discoverStructure(graph)            -> CascadeAnalysis
  +-- computeCounterfactual(graph, ...)   -> EpochSnapshot[]
  +-- scanTailRisk(shocks)                -> OmegaState
  +-- computeDoomsday(shocks, buffer)     -> DoomsdayState
  +-- validateSnapshot(snapshot)          -> TarskiValidationResult
  +-- solveInterdiction(graph, ...)       -> InterdictionResult

SPIRTES Structure Discovery

Causal structure discovery using constraint-based and score-based algorithms.

  • DCD (Directed Cyclic Discovery) — primary structure learning
  • PCMCI+ — temporal lag analysis for time-series causation
  • FCI — latent confounding detection under hidden variables

The Trinity Panel visualizes all three discovery methods as SVG circular layouts, allowing comparison of which edges each method identifies.

Cascade Analysis: Computes spectral radius (lambda_max) of the weighted adjacency matrix. If lambda_max < 1.0, the system is stable. Top-3 centrality nodes are identified by degree centrality. Displayed as: dS/dt = -damping*S + forgettingRate

TARSKI Truth Verification

Physical constraint validation using a formal axiom library. In v3, Tarski also validates SystemStateSnapshot objects before they enter the store.

Axiom Levels

LevelTypeCountExamples
L0Physics (immutable)6Temporal Priority, Conservation of Flow, DAG Integrity, Carnot Bound, Shannon Capacity
L1Regulatory (red alert)4Sanction Logic (OFAC/EU), Force Majeure, Capital Adequacy, EUV Export Control
L2Heuristic (anomaly flag)2Lead-Lag Reversal, Capacity Saturation

Programmatic Axiom Checks (Snapshot Validator)

CheckAxiomWhat it validates
Temporal PriorityA-01No negative edge weights (reversed causality)
Conservation of FlowA-02Node omega does not exceed theoretical max (10)
DAG IntegrityA-03Self-loop / degenerate cycle detection

In VERIFIED mode, restricted nodes display a red badge and inconsistent edges render as red dashed lines. Proof traces show which axioms were violated and the check result. All validation is currently programmatic (no external SMT solvers) — Z3/cvc5 integration is planned for v2 backend.

PEARL Causal Reasoning

Pearl's do-calculus for counterfactual reasoning and graph surgery. The Pearl module now includes three sub-tools: Intervention, Ablation, and Interdiction.

Intervention Mode (do(X))

When a target node is selected, the system computes:

  • Expected Utility: E[U] = target.omega x avg_downstream_weight x 10
  • Regret Bound: (1 - avg_weight) x target.omega
  • Affected Nodes: BFS downstream with delta-omega estimates
  • Method: DeepCFR (if confounded paths) or BackdoorAdjustment

Scissors Tool (Graph Surgery)

Cutting an edge triggers severEdgeAndSpawnConsequences():

  1. Edge marked as severed (red, dashed, 25% opacity)
  2. 2 consequence nodes spawned from domain-specific templates (omega 7.5–9.5)
  3. New edges connect consequences to the original edge's target (weight 0.8)
  4. Consequence nodes render orange with birth animation

Ablation Analysis

Select nodes/edges for removal and compare the resulting cascade against the baseline. Located in AblationPanel.tsx + ablation-engine.ts.

  • Toggle nodes/edges for ablation (auto-ablates connected edges when a node is selected)
  • computeAblationComparison(graph, nodeIds, edgeIds) computes structural delta
  • "Run" builds the ablated graph and launches cascade replay on the intervention timeline
  • Mutually exclusive with scissors mode (enabling one disables the other)

Interdiction Solver

Greedy minimax network protection at interdiction-engine.ts + InterdictionPanel.tsx.

  • Algorithm: O(budget x candidates) greedy approximation to NP-hard network interdiction
  • Budget: configurable (default 3) — max number of removals
  • Mode: edge-only, node-only, or both
  • Damage metric: composite of peak mean shock intensity (0-40), omega buffer deficit (0-40), and criticality bonus (0 or 20)
  • Requires active shocks to run; outputs ordered list of optimal interventions with marginal reduction scores

07. Snapshot System

The snapshot system provides a structured data frame (SystemStateSnapshot) that captures the entire graph state at a point in time. Snapshots are built from engine outputs, validated by Tarski, and stored in the Zustand store for the copilot to reference.

SystemStateSnapshot Schema

{
  version: 1,
  timestamp: string (ISO-8601),
  graph: {
    nodes: [{ id, omega, fragility, isActivated }],
    edges: [{ id, weight, probability, isSevered }]
  },
  engineOutputs: {
    spirtes: { density, lambdaMax, isStable } | null,
    pareto:  { omegaBuffer, status, criticalityEstimate } | null,
    pearl:   { interventionCount, severedEdges[] } | null
  },
  tarskiValidation: {
    status: "PENDING" | "PASSED" | "VIOLATIONS_FOUND",
    violations: [{ axiomId, edgeId?, nodeId?, detail }],
    checkedAt: string
  },
  metadata: { epochCount, shockCount, activeModule }
}

Snapshot Lifecycle

SNAPSHOT PIPELINE
  Graph state + Engine outputs
      |
      v
  buildSnapshot() (serializer.ts)
      |-- Derives node states from graph or active epoch
      |-- Computes cascade analysis (lambda_max)
      |-- Computes omega state from shocks
      |-- Assembles pearl intervention data
      |
      v
  SystemStateSnapshot (tarskiValidation = PENDING)
      |
      v
  setSnapshot() in Zustand store
      |-- validateSnapshot() (tarski-validator.ts)
      |   |-- Runs 5 axiom checks (A-01, A-02, A-03, H-02, R-01)
      |   +-- Returns PASSED or VIOLATIONS_FOUND
      |
      |-- Stores with validation result attached
      |-- Appends to snapshotHistory (capped at 50)
      +-- Console warns on violations (informational in v3)

  Between snapshots:
  diffSnapshots(prev, curr) -> SnapshotDiff
      |-- Node diffs: omega delta, fragility delta, activation changes
      |-- Edge diffs: weight delta, severance changes
      |-- Added/removed nodes and edges
      +-- Metric diffs: omega buffer delta, stability changes, violation count

File Structure

  src/lib/snapshots/
  +-- types.ts                 SystemStateSnapshot, SnapshotNode/Edge, validation types
  +-- serializer.ts            buildSnapshot() — pure function, no API calls
  +-- tarski-validator.ts      validateSnapshot() — programmatic axiom checks
  +-- diff.ts                  diffSnapshots() — structured delta between snapshots

08. Cascade Replay System

The replay system simulates cascade propagation across the causal graph over discrete time epochs, allowing users to visualize how shocks ripple through the network.

Cascade Simulation

simulateCascade(graph, shocks, severedEdges, config?, epochs?, initialNodeStates?) produces an array of EpochSnapshot objects, each containing per-node shock states, omega buffer, and criticality status.

Dual Timeline Architecture

TimelineSourcePurpose
baselinestartReplay()Cascade with current graph + shocks + severed edges
interventionbranchFromCurrentEpoch() or startAblationReplay()Cascade from a branch point or ablated graph

Replay Controls

  • Play / Pause: Toggle epoch auto-advance
  • Speed: 0.5x, 1x, 2x, 4x (driven by useReplayTick hook)
  • Epoch Slider: Scrub through cascade timeline
  • Step: Advance/rewind by one epoch
  • Timeline Toggle: Switch between baseline and intervention views
  • Branch: Fork a new intervention timeline from the current epoch

Dataset Tracking & Removal

Always merge, never replace. Imported data is added on top of the existing graph. Each import is tracked as an ImportedDataset with a unique color from the 8-color palette. Datasets can be removed individually, which strips their nodes, edges, and any orphaned color-matched nodes not claimed by other datasets.

Edge Inference (Enrichment)

enrich.ts: inferEdges() automatically creates directed edges between imported nodes based on shared domain (sequential chaining) and shared category (cross-domain bridges). This prevents imported clusters from remaining fully disconnected.
